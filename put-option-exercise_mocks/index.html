Pricing Engine — YTP Computation NEW BE LOGIC
How the backend computes purchase_price for put-option bonds. approved_ytp is fixed once at bond onboarding (from treasury/ops approval); purchase_price is recomputed every morning as settlement_date advances — guaranteeing the displayed yield always equals the approved rate exactly.
Computation Flow
📥 INPUT — Bond Master DB
approved_ytp  15.0%  SET ONCE AT ONBOARDING
put_option_date  e.g. "2026-05-15"
coupon_rate  13.50% p.a.
payment_frequency  Monthly
face_value  ₹1,00,000
r  = 1.1715%  CONSTANT — computed once from approved_ytp, stored
Dynamic — recalculated daily / at checkout
settlement_date  T+1 or T+2
📅 Before 3:00 PM on business day → T+1
📅 At/after 3:00 PM or weekend/holiday → T+2
↓
⚙️ COMPUTE — Reverse discounting to derive purchase_price (runs daily)
1
r is already stored as a constant  →  skip recomputation
r = 1.1715% was derived once at onboarding: (1 + 15/100)^(1/12) − 1
2
Generate coupon schedule from settlement_date → put_option_date only  →  discard all cashflows after put date
3
Append face_value as the final cashflow at put_option_date  →  principal returned at put, not maturity
4
Discount all cashflows at rate r  →  purchase_price is the output
purchase_price = Σ [coupon_t / (1+r)^t] + [face_value / (1+r)^n]  →  ₹99,523
↓
📤 OUTPUT — Two prices produced
Listing Display Price (daily cron)
Settlement = T+1 (indicative)
Used on: bond card, explore page
₹99,523  pulls to par daily
Checkout Price (at trade)
Settlement = actual T+1 or T+2
Used on: confirmation screen
₹99,523 (T+1)  /  ₹99,562 (T+2)
YTP = 15.0% in both cases
yield_to_put = 15.0  (always = approved_ytp)
effective_yield = 15.0
effective_maturity_date = "2026-05-15"
has_put_option = true
↓
🔄 DOWNSTREAM — Fields flow into
Listings Detail API
/listings/{id}/details
Listing Feed
/bff/api/v1/bond-listing
Sort / Filter Engine
sort_by=ytm uses effective_yield
Cashflow Endpoint
?scenario=put truncates at put_date
Daily Price Drift — Pull to Par
As settlement_date advances, coupon periods shrink. To maintain exactly 15% YTP, purchase_price rises toward par. Only n changes — r stays constant at 1.1715%.

Edge Cases
ConditionBehaviour
put_option_date in the pastSkip computation; has_put_option = false; alert Ops
Put price ≠ face valueUse actual put price as final cashflow (confirm Open Q1)
Multiple put dates (Bermudan)V1: use earliest put date only
put_option_date not setNo computation; bond displays as normal YTM bond
approved_ytp not setBlock bond from listing; alert Ops — cannot derive purchase_price without target yield
