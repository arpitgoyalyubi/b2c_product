Section 2

Core Sell Price Formula

When a user initiates a sell, the backend computes the price in 5 steps. No manual input from treasury or ops.

  1. Identify remaining cashflows

    Fetch the bond's full coupon schedule from bond master. Filter to all coupons after the sell date, plus the principal repayment at maturity.

  2. Compute Aspero's discount rate

    aspero_rate = bond_coupon_rate + aspero_spread_bps

    e.g. 12% + 2% = 14%/year. Convert to per-period rate: 14% ÷ 12 = 1.1667%/month

  3. Discount each cashflow to present value

    PV = cashflow_amount ÷ (1 + monthly_rate) ^ months_to_payment

    Sum all PVs. Use the annuity formula for equal recurring coupons (see Section 3).

  4. Add accrued interest

    AI = face_value × coupon_rate × (days_since_last_coupon ÷ 365)

    Compensates the user for coupon earned but not yet paid. Zero if selling exactly on a coupon date.

  5. Sell price

    Sell Price = Sum of PVs + Accrued Interest

    This is the per-unit price shown to the user on the sell initiation screen before confirmation.

// Core sell price formula sell_price_per_unit = pv_of_coupons + pv_of_principal + accrued_interest // PV of recurring monthly coupons — annuity formula pv_of_coupons = C × [1 - (1 + r)-n] / r // PV of lump-sum principal at maturity pv_of_principal = face_value / (1 + r)n // Accrued interest since last coupon date accrued_interest = face_value × coupon_rate × (days_since_last_coupon / 365) // Where: // C = monthly coupon amount (face_value × annual_rate ÷ 12) // r = monthly discount rate (aspero_rate ÷ 12) // n = number of remaining monthly coupons from sell date
📌
Floor guardrail: If the computed sell price drops below min_sell_price_floor_pct × face_value (default 85%), cap the sell price at the floor. This prevents extreme cases with very high spreads or unusual market conditions.
Section 3

Formula Deep Dive — Why Two Separate PVs?

The bond produces two structurally different cashflows. They need separate discounting treatment.

PV of Coupons

What it is: 35 identical ₹10 payments, each at a different future month.

Why annuity formula: Discounting each individually would need 35 calculations. The annuity formula collapses them into one — same result, one step.

C × [1 − (1+r)−n] / r
= 10 × [1 − (1.01167)−35] / 0.01167
= 10 × [1 − 0.666] / 0.01167
= 10 × 28.6
= ₹286

PV of Principal

What it is: A single ₹1,000 payment at month 35 (maturity). The annuity formula doesn't handle it — it only handles recurring equal payments.

Simple single-payment PV:

face_value / (1+r)n
= 1000 / (1.01167)35
= 1000 / 1.501
= ₹666

Intuition: Why Does PV of Principal Change So Much?

The principal is the same ₹1,000 regardless of when the user sells. But how far away it is changes drastically:

Sell atMonths to principalPV of ₹1,000Difference vs face
Day 3035 months₹666−₹334
Day 9033 months₹682−₹318
Day 36524 months₹757−₹243
Maturity0 months₹1,000₹0

This "pull to par" — principal converging toward ₹1,000 as maturity approaches — is the dominant driver of sell price increasing over time.

(2) Fewer discounted cashflows — each passing month is one less future coupon being discounted at Aspero's penalising 14% rate. Both push the sell price toward ₹1,000 as maturity approaches.
Section 6

Record Date Edge Case

The record date determines who receives a coupon. Since Aspero settles on T+1, the sell date relative to the record date changes who gets the upcoming coupon — and must change the sell price accordingly.

How It Works

TermDefinition
Record dateBondholder registered on this date receives the next coupon
Ex-coupon dateRecord date minus 1 trading day (given T+1 settlement)
Cum-couponSelling before ex-date → buyer (Aspero) gets next coupon → seller gets AI in sell price
Ex-couponSelling on/after ex-date → seller stays on record → seller gets coupon directly → AI must NOT be added to sell price for that period

Concrete Example

// Coupon 3 timeline Record date = Day 89 // must be registered by here Coupon paid = Day 90 // ₹10 sent to whoever was on record Day 89 // Case A — Sell on Day 87 (before ex-date) // Settlement = Day 88. Aspero on record Day 89 → Aspero gets Coupon 3 days_since_last_coupon = 87 - 60 = 27 days accrued_interest = 1000 × 12% × (27/365) = ₹8.9 ← paid to user // Case B — Sell on Day 89 (on ex-date) // Settlement = Day 90. User still on record Day 89 → User gets Coupon 3 directly accrued_interest = ₹0 ← user gets coupon separately, AI must NOT double count // Also: exclude Coupon 3 from remaining cashflows in PV calculation
🚨
Engineering note: The sell price engine must check: if sell_settlement_date >= record_date_for_next_coupon → exclude the next coupon from PV cashflows AND set AI = 0. Failing to handle this results in the user being paid twice for the same coupon period.

Section 7

Post-Record Date Purchase

If a user buys a bond after the record date for an upcoming coupon, they don't receive that coupon — and Aspero sells the bond to them at a lower "ex-coupon" price. After the 30-day hold, here's how the sell price changes.

Example Timeline

Section 8

Configuration Parameters (Phase 1 — Global Spread)

All parameters are set via the Seller Portal. No app release required to change pricing.

ParameterDefaultEffect on sell price
aspero_spread_bps TBD (finance to align) Higher spread → lower sell price (more Aspero margin). Lower spread → higher sell price (less margin). Applied to all ISINs globally.
day_count_convention Actual / 365 Used in accrued interest formula. Standard for Indian corporate bonds.
min_sell_price_floor_pct 85% of face value Hard floor. If computed sell price < 85% of face, cap at ₹850 (for ₹1,000 face). Risk guardrail — prevents extreme pricing in edge cases.
min_hold_days 30 days Sell CTA is disabled until this many days have elapsed since settlement date.
✅
All formula inputs are already in bond master at onboarding: face_value, coupon_rate, coupon_frequency, maturity_date, full cashflow schedule. No new data collection needed. The sell price engine reads these at sell initiation time.
