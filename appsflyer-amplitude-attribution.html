What we're trying to achieve

Goal

Attach AppsFlyer attribution data — media source, campaign, channel, organic vs paid — to every user in Amplitude, so that all downstream behaviour (signup, KYC, first invest) can be segmented by acquisition channel.

Example questions this should answer: "What is the KYC completion rate for users coming from Google Ads vs organic?" or "Which campaign produces users with the highest first-trade rate?"

Important: The goal is attribution properties on Amplitude users — not just the [AppsFlyer] Install event count. Even if the install event is missing, as long as attribution user properties (media_source, campaign, channel) are set correctly, all downstream analysis works.

How AppsFlyer measures installs

Install = first app open, not store download

AppsFlyer counts an install when the app is opened for the first time after being downloaded. The download itself is not tracked — the first launch is the trigger.

Sequence on first open

Native layer starts Native
AppsFlyer native SDK initialises immediately — before React Native loads. Sends install signal to AF servers.
AppsFlyer servers do attribution lookup AF Servers
Match this device against recent ad clicks/impressions. Resolve: media source, campaign, channel. This takes 2–5 seconds.
React Native loads JS Layer
500ms–3s after native start, depending on device. App.tsx mounts, analytics hooks run.
Attribution callback fires on device AF → Device
onInstallConversionDataLoaded delivers { media_source, campaign, af_channel, af_status } back to the app.
S2S postback fires AF Servers → Amplitude
Separately, AF servers also attempt to forward the install event to Amplitude's servers directly — requires Amplitude device ID to be attached to the AF user record.
Key point: There are two independent paths for attribution data to reach Amplitude — the in-app callback (device-side) and the S2S postback (server-to-server). Currently only the S2S path is used, and it is partially broken.

Root cause analysis

Root Cause 1 — Structural timing gap

AppsFlyer's native SDK fires the install event before React Native loads. The Amplitude device ID (needed for S2S matching) is only available after the JS layer initialises. For users who close the app before RN loads, the S2S event is sent with no Amplitude ID → dropped.

Root Cause 2 — In-app callback not handled

onInstallConversionDataListener: true is set but appsFlyer.onInstallConversionData() handler is never registered. Attribution data (media source, campaign, channel) arrives in-app and is silently discarded — never written to Amplitude user properties.

Root Cause 3 — iOS has no GAID fallback

On Android, GAID (Google Advertising ID) allows probabilistic matching between AF and Amplitude — both read it natively. iOS has no equivalent without ATT consent (most users deny). Result: iOS S2S capture is ~6% vs Android ~59%.

Root Cause 4 — requestIdleCallback adds extra delay

Analytics init is deferred until the browser/JS engine is idle. On low-end Android devices this adds an additional 1–2 seconds before setAdditionalData is called, increasing the window where AF sends S2S events without an Amplitude ID.

Fix 1 — In-app attribution callback Recommended
Solves Root Causes 2, 3, 4 · ~15 lines in AppsFlyerService.ts · Low effort · High impact

How it works

Register appsFlyer.onInstallConversionData() after initSdk. When AppsFlyer's attribution lookup completes (2–5s after first open), this fires on the device with full attribution data. At that point React Native is loaded and Amplitude is initialised — write the attribution directly to Amplitude user properties with identify().

This path is entirely device-side. It does not depend on S2S, does not need the Amplitude device ID for server matching, and does not depend on timing relative to AppsFlyer's postback queue.

Guard required: check Amplitude device ID before calling identify()
Even though the current init order in useAppLaunchConfig.ts guarantees AppsFlyer only starts after AmplitudeService.getAmplitudeIds() resolves, the callback itself has no built-in knowledge of this ordering. If the init flow is ever refactored, or if Amplitude SDK fails to assign a device ID, calling identify() on an uninitialised SDK could attach attribution to the wrong user or produce a no-op write that overwrites a future correct identity.

Rule: call getDeviceId() at the top of the callback. If it returns undefined, return early — the attribution data is already being sent via S2S fallback and should not be double-written via a potentially uninitialised SDK.

Why this is reliable

No server-to-server dependency — attribution data goes device → Amplitude SDK directly, not through AF servers → Amplitude servers.
No device ID matching required — the Amplitude SDK is already running on the device when the callback fires. User is already identified.
Fires after RN is loaded — attribution callback resolves 2–5s after first open, by which time React Native and Amplitude SDK are always ready.
Works on iOS without ATT — does not rely on IDFA. Attribution data is returned from AF servers for all users (organic = "Organic", paid = media source name).
Fires only once per install — subsequent app opens do not re-fire this callback. Attribution user properties are set once and persist on the Amplitude user profile.

Code change — src/manager/analytics/appsFlyer/AppsFlyerService.ts

// Inside requestATTAndInitAppsFlyer(), after appsFlyer.initSdk()

// Update existing import at the top of the file (getDeviceId already imported — add Identify, identify):
import { getDeviceId, getSessionId, Identify, identify } from '@amplitude/analytics-react-native';

// After initSdk success callback, add:
appsFlyer.onInstallConversionData((res) => {
  try {
    const d = res?.data ?? {};

    // ✅ Guard: only write to Amplitude if SDK has assigned a device ID.
    // AppsFlyer SDK initialises only after getAmplitudeIds() resolves (Phase 2
    // in useAppLaunchConfig), so this will be defined in normal flow.
    // Guard protects against future init order changes or SDK init failures.
    const deviceId = getDeviceId();
    if (!deviceId) return;

    const identifyObj = new Identify();

    // Core attribution properties
    if (d.media_source)  identifyObj.set('af_media_source', d.media_source);
    if (d.campaign)      identifyObj.set('af_campaign',     d.campaign);
    if (d.af_channel)    identifyObj.set('af_channel',      d.af_channel);
    if (d.af_status)     identifyObj.set('af_status',       d.af_status);   // "Organic" | "Non-organic"
    if (d.af_adset)      identifyObj.set('af_adset',        d.af_adset);
    if (d.campaign_id)   identifyObj.set('af_campaign_id',  d.campaign_id);

    identify(identifyObj);
  } catch (_) {
    // Silent — never block app flow
  }
});

Fix 2 — Improve S2S instrumentation Supplementary
Partially closes the install event gap · Does not replace Fix 1 · Medium effort
Note: Fix 2 improves the [AppsFlyer] Install event count in Amplitude but does NOT solve the attribution properties gap on its own. Fix 1 should always be implemented. Fix 2 is supplementary if you also need accurate install counts in Amplitude for funnel base calculations.

Why the S2S gap exists

AppsFlyer's native SDK fires the install event before React Native loads. The S2S postback to Amplitude is sent from AF's servers — it needs the Amplitude device ID attached to the AF user record. setAdditionalData({ AmplitudeDeviceId }) can only be called from JS, which loads after native. For users who close the app quickly, the postback fires before the device ID is set.

Option A — Move AppsFlyer native SDK init to native layer (High effort)

Initialize AppsFlyer at the native layer (AppDelegate.swift / MainApplication.kt) rather than from JS. This eliminates the RN load dependency entirely, but requires storing the Amplitude device ID natively (AsyncStorage is not accessible from native code on first open).

Option B — Use AppsFlyer attribution callback for Amplitude device ID (Medium effort)

Since the attribution callback fires after RN is loaded (2–5s), use it as a second opportunity to call setAdditionalData with the Amplitude device ID — ensuring the device ID is set before any subsequent AF events are forwarded.

// Inside the onInstallConversionData handler (same callback as Fix 1)
appsFlyer.onInstallConversionData((res) => {
  try {
    // ✅ Guard (Fix 1 already adds this — deviceId is confirmed defined here)
    const deviceId = getDeviceId();
    if (!deviceId) return;

    // Fix 1: set attribution properties on Amplitude
    // ... (see Fix 1 above)

    // Fix 2 Option B: re-confirm Amplitude IDs on AF for S2S linkage.
    // deviceId is already checked above — safe to pass directly.
    const sessionId = getSessionId();
    appsFlyer.setAdditionalData({
      AmplitudeDeviceId: deviceId,
      ...(sessionId !== undefined && { AmplitudeSessionId: sessionId }),
    }, () => {});
  } catch (_) {}
});

Option C — Enable IDFV matching in AppsFlyer dashboard (Low effort, iOS only)

In AppsFlyer's Amplitude integration settings, enable IDFV (Identifier for Vendor) as a matching method. IDFV is available on iOS without ATT consent for apps from the same vendor. This can partially recover iOS install capture (currently 6%) without any code change.

Path: AppsFlyer Dashboard → Configuration → Integrated Partners → Amplitude → Attribution → enable IDFV matching.
