CKYC-Primary KYC Redesign

Moving POI / POA capture from DigiLocker-primary to a CKYC-first hybrid — with DigiLocker as fallback and manual upload + Video IPV as the final safety net.

Proof of Identity · Proof of Address ~70–80% projected DigiLocker bypass 2–3 week integration estimate Status: Solutioning

Today

DigiLocker is primary — the sole automated source for both PAN (POI) and Aadhaar address (POA). No manual POA path for individuals.

→

Proposed

CKYC is primary — ~70–80% of KYC-validated users skip DigiLocker via a mobile-based CKYC search. DigiLocker becomes the fallback.

1Why this change

The current DigiLocker-primary flow is the single sharpest leak at the top of the KYC funnel, with no graceful degradation for address proof.

Current behaviour (verified against the yubi-b2c-mobile repo + live Amplitude, 8-week window ending 03 Jul 2026):

  • DigiLocker sits at step 1 and fetches PAN and Aadhaar address together in one callback.
  • The fallback is asymmetric and broken on the POA side:
    • PAN fetch fails → user adds PAN manually → verified directly against Income-Tax (ITR) records (name/DOB match). recoverable ~67%
    • Aadhaar fetch fails / consent denied → no manual POA upload path for individuals. User can only retry DigiLocker. recovery ~38%

Funnel leakage at this step:

SignalMagnitudeMeaning
Never click DigiLocker CTA~26%Of users who start KYC, ~1 in 4 never attempt DigiLocker.
Abandon inside DigiLocker~22%Of SDK launches, callback returns for only ~78% — consent confusion / UIDAI or DigiLocker outage.
Aadhaar fetch failure (within callback)~7%Data unavailable; only ~38% of these recover.
PAN fetch failure (within callback)~8%Routed to manual PAN → ITR.

Net: DigiLocker is a hard dependency that single-handedly gates the top of the KYC funnel, and address proof has no manual escape hatch. CKYC-primary removes that dependency for the majority and gives POA a graceful fallback for the first time.

3Proposed end-to-end UX flow

Three tiers of assurance: authenticated-source (CKYC) → authenticated-source (DigiLocker) → manual upload with a compensating Video IPV.

STEP A · PRIMARY

CKYC search

~70–80%
  • No manual mobile entry — we already hold the user's mobile from signup, so the CKYC search auto-fires on it in the background (see O1 / O8).
  • User sees a single consent screen — "We found your existing KYC. Confirm to securely fetch it." The consent tap is mandatory (regulatory).
  • CERSAI sends an OTP to the registered mobile → auto-read where possible (Android SMS Retriever), manual entry as fallback (see O9).
  • Download the CKYC record: demographics + address (POA) + PAN (POI) + photo.
  • Capture a live selfie → face-match against the CKYC photo to bind the record to the real person.
  • Broker enhancement: for stock-broker records, Digio can pull the actual Aadhaar & PAN images from the account-opening form stored at the CKYC end — not just the CKYC JSON.
✅ found · current · face match → skip DigiLocker entirely  |  ❌ not found / mobile mismatch / stale / inadequate record → fall through
STEP B · FALLBACK

DigiLocker

~20–30%
  • The existing Hyperverge / DigiLocker SDK journey, now demoted to fallback — behaviour unchanged from today.
  • Fetches Aadhaar (POA) + PAN (POI).
✅ success → continue  |  ❌ fail / consent denied / data unavailable → fall through
STEP C · SAFETY NET

Manual upload + Video IPV

  • POA = Aadhaar only; POI = PAN only (D1). Any other document type throws an error.
  • OCR extracts Name / PAN / DOB → validated against Income-Tax records.
  • On name/DOB mismatch → error, with a configurable cap on re-attempts before terminating the journey.
  • Because Aadhaar is supplied manually, a mandatory Video IPV replaces the selfie for this cohort (see §4).

Downstream sequence is unchanged — only the POI/POA capture at the front changes:

Signup / PIN› Risk Profiling› POI/POA — CKYC → DigiLocker → Manual+VIPV› Bank› Liveness*› Nominee› Wet signature› Aadhaar e-Sign› Demat

* For the CKYC and manual-Aadhaar cohorts the "liveness" role is served by the CKYC face-match selfie / VIPV. Whether the standalone step-3 liveness selfie still runs for those cohorts is an open product decision (see O4).

4Video IPV — what it is, and why

The mechanics (self-serve, no agent)

  • System displays a randomly generated code / OTP on screen and starts a timer.
  • User records themselves reciting the code to the camera.
  • System captures both the full video and a still image of the customer.
  • Media is stored tamper-proof with a date/time stamp as proof of IPV.

Is it additional over the current liveness check?

No — it is a substitution, and only for the manual-Aadhaar cohort. VIPV replaces the still-selfie liveness capture for users who upload Aadhaar manually. Everyone else keeps the lighter check.

CohortIdentity / liveness check
CKYC path ~70–80%Live selfie face-matched against the CKYC photo
DigiLocker fallbackExisting still-photo liveness selfie (as today)
Manual Aadhaar uploadVIPV replaces the selfie

Why VIPV is required for manual upload specifically: The other two paths pull the document from an authenticated source — DigiLocker is authenticated by UIDAI, CKYC is a KYC-verified record from CERSAI. The source itself vouches that the document is genuine and belongs to that identity.

A manually uploaded Aadhaar image has no such guarantee. OCR + ITR validation only proves the document is internally consistent — it does not prove the person holding the phone is the document's owner (a genuine Aadhaar that isn't theirs, a doctored image, or a photo-of-a-photo would all pass). A still selfie is easily spoofed with a printed photo.

VIPV is the compensating control: a random on-screen code + liveness + timer + tamper-proof time-stamped capture proves a live human — not a pre-recording — is present and consenting, and binds that face to the uploaded document. This is also the SEBI-expected standard for the manual/offline-OVD path (random action, liveness, geo-tag for India presence, tamper-proof storage).

Rule of thumb: stronger source → lighter check (selfie / face-match); no trusted source → stronger check (VIPV).

6Compliance considerations

  • CKYC "last-updated date": every CKYC record carries a last-updated timestamp. We must set the staleness threshold beyond which we reject the CKYC record and fall back to DigiLocker. Action: Arpit to confirm internal compliance policy on acceptable record age.
  • VIPV regulatory basis: SEBI permits self / app-based Video In-Person Verification with random code recital, liveness check, time-stamping, geo-tagging (physical location in India) and tamper-proof storage. Digio's VIPV must meet these — confirm geo-tagging is captured.
  • CKYC as a valid POI/POA source: CKYC (a 14-digit KIN maintained by CERSAI) is a recognised, reusable KYC source across regulated entities in the securities market.

References: SEBI clarification on online/video/app KYC · SEBI circular on tech in KYC · CERSAI · CKYC guide.

8Open questions / risks

Not yet resolved in the meeting — these should be closed before committing to build. The ~70–80% and the whole CKYC-primary design remain Digio's proposal, not yet validated on our data.

O1CKYC hit-rate is an assumption. The ~70–80% is Digio's estimate. We can't validate it today — there is an instrumentation gap (no distinct CKYC-search event). Add CKYC-search success/fail/not-found events before launch to measure the real hit-rate.
O2Face-match source of truth. CKYC provides a photo; matching needs a live selfie. Confirm whether the CKYC path still requires our existing liveness selfie, and whether that double-counts with VIPV for manual users.
O3CKYC photo quality. Archive photos can be old / low-res, risking false face-match rejects on valid records. Define the fallback when face-match fails on a genuine CKYC record (retry → DigiLocker → VIPV?).
O4Liveness vs VIPV overlap. Today's step-3 liveness selfie may be redundant for CKYC/VIPV cohorts. Decide whether to keep, skip, or merge to avoid an extra friction step.
O5VIPV drop-off risk. Recording yourself reading a code is materially higher-friction than a selfie. It hits only the small manual cohort, but instrument its completion rate.
O6Mobile-number mismatch. CKYC search is keyed on the registered mobile. If the app mobile ≠ CKYC-registered mobile, search fails → fallback. Needs an "edit number / not you?" affordance. Quantify how often this happens.
O7e-mandate / eNACH is not part of this KYC flow — flagged as out of scope so it isn't conflated.
O8In-context UI during the background CKYC hit. With manual mobile entry removed, we must show the user context while we auto-query CKYC ("Checking your existing KYC…") so they aren't lost. This should be an Aspero-owned screen, not the SDK default. Open: confirm with Digio whether the CKYC-search window is inside their SDK (their UI) or a headless API call we wrap in our own screen — we want the latter.
O9OTP auto-read feasibility. Possible but not guaranteed — Android SMS Retriever needs the CERSAI OTP SMS to carry the app hash and land on the same SIM/device; iOS is keyboard-autofill only. Open: confirm the CERSAI OTP sender format with Digio; keep manual entry as fallback regardless.
