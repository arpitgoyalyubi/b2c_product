Section 3 — Design Recommendations (from screen review)
🎨

Screen-by-screen UX issues & fixes

Production screenshots reviewed. Issues ranked P0 (do now) / P1 (next sprint) by estimated drop-off impact.

Q1 — Occupation
Screen 1 What is your occupation? 4 issues
P0
Verbose intro text hides MCQ tiles on load
The "It is always better to know each other…" subtitle is 27 words. On iPhone SE and budget Android, this pushes all 8 tiles off-screen — users see only text. Likely the primary cause of 1,514 users (38.3%) never tapping any option.
Replace with 1-line max: "Help us personalise your experience". Move the copy to a collapsed "Why we ask ?" tooltip. All tiles become visible above the fold.
P0
Most-common answers buried in row 3
Current order: Service → Student → Housewife → Landlord → Business → Professional. Business and Professional are the most-selected occupations yet appear in row 3 — below the fold on small screens.
Reorder by selection frequency: Business · Professional · Service · Housewife · Landlord · Student · Agriculture · Others. Top answers visible immediately on load.
P1
Large icon tiles force scrolling through 4 rows
Each tile is ~130px tall. 8 tiles × 2 columns = 4 rows × 130px ≈ 520px of grid, plus intro text. On a 667px viewport the last row is never visible without scrolling.
Reduce tile height to ~80px (smaller icon, label below). All 8 tiles fit on-screen in a single view on most phones. Or use a 3-column compact grid.
P1
"Housewife" label is dated and non-inclusive
Urban and metro users — and male homemakers — may hesitate to self-identify with this term.
Rename to "Homemaker." One-word change, zero logic impact.
Q2 — Annual Income
Screen 2 What is your annual income? 1 issue
P1
"Annual" not clarified — users may enter monthly income
Tier-3/4 users and daily-wage earners commonly think in monthly terms. If a user picks "Below 1 lakh" meaning ₹8,333/month, the answer is wrong — corrupting persona data and downstream bond recommendations.
Add grey subtext beneath the question: "Per year, before tax." Also add a "Why we ask ?" tooltip: "We use this to suggest bonds that match your investment capacity." No layout change needed.
Q3 — Net Worth · biggest dropout
Screen 3 What is your net worth? −1,323 users (30.3%) · 4 issues
P0
"Net worth" is undefined — users stall and abandon
Most retail users don't know whether net worth includes their home, PF balance, gold, or car. No tooltip or definition exists. Users who can't answer in ~5 seconds abandon — this explains the majority of the 1,323 dropout at this step.
Add a collapsible hint: "Net worth = total assets (savings, investments, property) minus liabilities (loans, debts). Approximate is fine." ASP-3817 scope — ~30-min code change with large funnel impact.
P0
Bottom bucket starts at ₹10 lakh — too high
Young users (22–28, P6 persona) with net worth below ₹1 lakh see no matching option. They abandon rather than pick an incorrect bucket.
Add "Below 1 lakh" as the first option, rename current "Below 10 lakh" → "1 to 10 lakhs." Requires a data migration note but directly unlocks this cohort.
P1
Upper ranges use lakh notation — hard to parse at scale
Top options read "100 to 500 lakhs" and "More than 500 lakhs." Indian users think in crores at this level (₹1 cr = ₹100 L). Mental conversion breaks the scan-and-tap flow.
Switch upper tiers to crore notation: "1 to 5 crore" / "More than 5 crore." Keep lower tiers in lakhs. Mixed notation matches how Indians naturally discuss wealth.
P1
Icons identical to income screen — question context lost
Q2 and Q3 use the same rupee coin/bag icons in the same 2-column layout. Skimming users can't visually tell they've moved to a different question.
Use distinct Q3 icons — piggy bank, house, or balance scale — to visually signal "assets" vs "income."
Q4 + Q5 + PEP
Screen 4 Father/Husband Name · Marital Status · PEP 2 issues · −702 users after Q3
P1
3 different input types on one screen create mixed signals
The screen combines a text input (Father/Husband name) + MCQ tiles (Marital status) + a checkbox (PEP). The Continue CTA is visible without scrolling, but the mix of paradigms signals "this is complicated" to a user already fatigued from 3 MCQ screens.
For a future sprint: move the PEP checkbox onto its own step so text + MCQ feel lighter. The current layout is functional — prioritise P0 fixes first.
P1
"Father / Husband name" purpose unexplained
No placeholder text or tooltip explains why this is collected. For unmarried women the label is ambiguous — "is this my father's or future husband's name?" Leads to wrong data or hesitation.
Add placeholder: "As per Aadhaar / PAN record" + "?" tooltip: "Required for SEBI KYC. Enter your father's name if unmarried." Zero logic change needed.

Non-completers fired more answer-selected events than a single 4-MCQ session allows. They are returning on new sessions, having answers wiped, and re-answering from Occupation (Q1) — never finishing.

💾

Answers are NOT persisted

Code-confirmed: every session starts from scratch

useRiskProfileData.ts — fetchProfilingData()

On mount, calls GET /user-profiling-service?questionType=RISK_PROFILING which returns unansweredQuestions. Every answer field is reset to null / '' / noValue — no AsyncStorage, no Hookstate draft, no partial-save API.

Impact

3,957 users viewed risk profiling but never completed (May 1–27). Of these, 1,514 (38%) never answered any question. The remaining 2,443 answered Occupation (Q1) and continued further — on return, answers are wiped and they restart from Q1 every session.

Section 5 — Bugs Found During Analysis
🐛

Two instrumentation bugs making analytics unreliable

Discovered during data analysis — both affect decision-making quality.

🔴 P0 — RISK_PROFILING_SCREEN_VIEWED fires 19–20× per user/day

Should fire once on mount. Instead fires on every re-render because postEvent is in the useEffect dependency array and is not memoised.

Sample (7 days): 10,232 events for 523 users = 19.6× ratio.

Makes "how many times did a user return" unanswerable from this event. Also 19× unnecessary Amplitude API calls per user.

useRiskProfilingV2.ts:32 Analytics integrity
🟡 P1 — onInputChange (Q4 Name field) has zero tracking

The text input for Father/Husband Name (onInputChange) fires no event at all — only onInputBlur fires RISK_PROFILING_FATHER_HUSBAND_NAME_ENTRY. Q4 blur event registered 3,439 unique users in May, but completers = 5,268 — meaning ~35% of completers bypassed the field or the blur event didn't fire for them (partial tracking).

Can't tell whether users are abandoning because they don't know this info or because the field itself has a UX issue.

useRiskProfileData.ts:48 Missing signal
🟡 P1 — RISK_PROFILING_VALIDATION_ERROR never fires

0 events in 30 days. Either client-side validation is never triggered (users can submit with missing fields), or the validation code path is broken. Given 2,035 API errors (backend rejection), some submissions are going to the server without client-side validation catching them first.

useRiskProfilingV2.ts:74 Validation gap
Fix for SCREEN_VIEWED overfiring:
// ❌ Current — postEvent in deps causes re-fire on every render useEffect(() => { if (!isLoading && questions?.length > 0) { postEvent(riskProfilingV2Events.screenView()); } }, [isLoading, questions?.length, postEvent]); // ← remove postEvent // ✅ Fixed — stable ref, fires exactly once }, [isLoading, questions?.length]);

ASP-3815 P0 · Bug Engineering · Analytics

Fix RISK_PROFILING_SCREEN_VIEWED overfiring (19–20× per user)

ASP-3816 P0 · Story Engineering · UX

Persist risk profiling answers to AsyncStorage between sessions

3,957 users viewed risk profiling but never completed (May 1–27). Non-completers average 7.87 answer selections vs a 5-question ceiling — proving they return to the form and have to start over. Save answers to AsyncStorage on each onSelectAnswer call. On mount, merge saved draft with API response before setQuestions(). Clear draft on successful submit.

useRiskProfileData.ts 3,957 affected users/month AsyncStorage draft
ASP-3817 P1 · Story Product · Design

Reduce Q2→Q3 drop-off (1,060 users/month abandon between Annual Income and Net Worth)

The Q2→Q3 transition is the biggest in-form drop-off — 26.8% of non-completers (1,060 / 3,957) abandon between Annual Income and Net Worth. Users know their income but can't easily recall exact net worth. Fix: (1) Add helper text "approximate is fine" or "your best estimate", (2) change from exact entry to range brackets (e.g. ₹5L–₹25L), (3) add a tooltip explaining what counts as net worth. Target: reduce Q3 drop by 40% = ~424 more completions per month.

Add session attempt tracking and fix missing Q4 event

(a) Add attempt_number property to RISK_PROFILING_SCREEN_VIEWED — read a per-user counter from AsyncStorage, increment on each mount, pass as event property. Enables measuring exact multi-session return rate once overfiring is fixed. (b) Add RISK_PROFILING_NAME_FIELD_FOCUSED event to onInputChange for Q4 (Father/Husband Name) — currently the name field has zero engagement data.
