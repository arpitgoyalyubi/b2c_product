Non-completers fired more answer-selected events than a single 4-MCQ session allows. They are returning on new sessions, having answers wiped, and re-answering from Occupation (Q1) — never finishing.

💾

Answers are NOT persisted

Code-confirmed: every session starts from scratch

useRiskProfileData.ts — fetchProfilingData()

On mount, calls GET /user-profiling-service?questionType=RISK_PROFILING which returns unansweredQuestions. Every answer field is reset to null / '' / noValue — no AsyncStorage, no Hookstate draft, no partial-save API.

Impact

3,113 users answered ≥1 question but never completed. The ~30% who abandon before Net Worth (Q3) answered Occupation + Annual Income correctly. On return, they answer both again before hitting Q3 again — wasted effort every session.

Section 4 — Bugs Found During Analysis
🐛

Two instrumentation bugs making analytics unreliable

Discovered during data analysis — both affect decision-making quality.

🔴 P0 — RISK_PROFILING_SCREEN_VIEWED fires 19–20× per user/day

Should fire once on mount. Instead fires on every re-render because postEvent is in the useEffect dependency array and is not memoised.

Sample (7 days): 10,232 events for 523 users = 19.6× ratio.

Makes "how many times did a user return" unanswerable from this event. Also 19× unnecessary Amplitude API calls per user.

useRiskProfilingV2.ts:32 Analytics integrity
🟡 P1 — onInputChange (Q4 Name field) has zero tracking

The text input for Father/Husband Name (onInputChange) fires no event at all — only onInputBlur fires RISK_PROFILING_FATHER_HUSBAND_NAME_ENTRY. Q4 registered only 13 events vs 6,000–8,000 for MCQ questions.

Can't tell whether users are abandoning because they don't know this info or because the field itself has a UX issue.

useRiskProfileData.ts:48 Missing signal
🟡 P1 — RISK_PROFILING_VALIDATION_ERROR never fires

0 events in 30 days. Either client-side validation is never triggered (users can submit with missing fields), or the validation code path is broken. Given 2,035 API errors (backend rejection), some submissions are going to the server without client-side validation catching them first.

useRiskProfilingV2.ts:74 Validation gap
Fix for SCREEN_VIEWED overfiring:
// ❌ Current — postEvent in deps causes re-fire on every render useEffect(() => { if (!isLoading && questions?.length > 0) { postEvent(riskProfilingV2Events.screenView()); } }, [isLoading, questions?.length, postEvent]); // ← remove postEvent // ✅ Fixed — stable ref, fires exactly once }, [isLoading, questions?.length]);

ASP-3815 P0 · Bug Engineering · Analytics

Fix RISK_PROFILING_SCREEN_VIEWED overfiring (19–20× per user)

Remove postEvent from useEffect dependency array in useRiskProfilingV2.ts:32. Currently the event fires on every re-render instead of once per screen mount, making all "visit count" analytics completely unreliable and adding 19× unnecessary Amplitude API calls.

useRiskProfilingV2.ts:32 Analytics integrity 1 line fix
ASP-3816 P0 · Story Engineering · UX

Persist risk profiling answers to AsyncStorage between sessions

3,113 users answered ≥1 question but never completed. Non-completers average 7.06 answer selections vs a 5-question ceiling — proving they return to the form and have to start over. Save answers to AsyncStorage on each onSelectAnswer call. On mount, merge saved draft with API response before setQuestions(). Clear draft on successful submit.

useRiskProfileData.ts 3,113 affected users/month AsyncStorage draft
ASP-3817 P1 · Story Product · Design

Reduce Net Worth question drop-off (1,323 users/month)

Net Worth (Q3) is the single biggest drop-off point — 30.3% of all non-completers abandon here. Users know their income but can't easily recall exact net worth. Fix: (1) Add helper text "approximate is fine" or "your best estimate", (2) change from exact entry to range brackets (e.g. ₹5L–₹25L), (3) add a tooltip explaining what counts as net worth. Target: reduce Q3 drop by 40% = ~530 more completions per month.
