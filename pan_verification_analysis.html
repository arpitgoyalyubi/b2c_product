Three independent drivers of the decline
Driver 1 · Primary · Structural
Fewer users click the DigiLocker CTA
74.3% → 66.1%
CTA Click% has declined -8.2pp over 8 weeks with no recovery. Something changed in the PAN screen UX between April and May that's causing more users to not engage. This is the single largest lever. ~152 additional users per week are now dropping off before even starting DigiLocker.
Driver 2 · Stable · Not worsening
DigiLocker abandonment (mid-flow drop)
17–20% stable
Users who open the DigiLocker SDK but never return have been flat at 17–20% throughout. iOS is an exception at 25–44% abandon (small volume, 38–58 users/week). This is a persistent structural loss but not a new problem driving the decline.
Driver 3 · Acute · May 18 double-hit
PAN not in vault + backup flow collapse
+53 extra lost users
In May 18: vault miss volume doubled (77→130 users) AND backup flow success halved (63.6%→43.8%) simultaneously. At the May 11 backup rate, 57 more users would have verified. This double-hit explains the -5.6pp single-week drop in May 18.

Recommended actions
Priority Action What it fixes Expected impact
P0 Investigate PAN screen UX change CTA Click% has declined 8.2pp since Apr 06. Identify any redesign, A/B test, or content change on the AADHAAR_PAN_INITIAL_SCREEN that went live between Apr 06–18. Recovering CTA% from 66% back to 74% = ~150 extra users/week entering the DigiLocker flow
P0 Investigate May 18 backup flow collapse Backup flow success dropped 20pp in one week (63.6%→43.8%) while vault miss volume doubled. Check if a new acquisition cohort is less document-ready, or if a UI/flow bug was introduced. Restoring to 64% success rate = ~26 additional verifications per week from vault-miss users
P1 Add pre-check prompt before DigiLocker launch ~15% of users who complete DigiLocker don't have PAN linked in their vault. A prompt — "Make sure your PAN is linked in your DigiLocker account before continuing" — reduces surprise drop-offs and may improve backup flow intent. Could reduce vault miss% from 15% → 8–10% for prepared users
P1 Fix iOS DigiLocker abandonment iOS users who open DigiLocker abandon at 33% vs 19% on Android. Small absolute volume (58 users/week) but the rate indicates a platform-specific SDK or UX issue. Bringing iOS abandon% to Android parity = ~8 additional verified users/week
P1 Fix AppsFlyer → Amplitude user property sync Campaign/adgroup attribution properties (gp:[AppsFlyer] campaign etc.) are in the Amplitude taxonomy but return no data for KYC events. Engineering needs to call amplitude.setUserProperties() with attribution data after the AppsFlyer SDK callback. Unblocks channel-level funnel analysis — critical for understanding which acquisition campaigns drive low-quality KYC cohorts
Ongoing Fix callbackResponse instrumentation The callbackResponse property on HYPERVERGE_DIGILOCKER_SDK_CALLBACK_RECEIVED always returns 0 — the HyperVerge SDK result (success/failure/cancelled) is not being passed to the Amplitude event. Would enable distinguishing real callback failures from user-cancelled flows
Event definitions
Funnel events (in order)
KYC_RISKDETAILS_SUBMIT — KYC started (anchor/denominator)
AADHAAR_PAN_INITIAL_SCREEN_START_DIGILOCKER_CTA_CLICK — CTA tapped
HYPERVERGE_DIGILOCKER_SDK_INITIATED — DigiLocker SDK opened
HYPERVERGE_DIGILOCKER_SDK_CALLBACK_RECEIVED — Returned from DigiLocker
DIGILOCKER_OUTPUT_API_SUCCESS_RESPONSE — Output API fired
KYC_PAN_VERIFIED — PAN verified (success endpoint)
KYC_PAN_FAILED — Hard PAN failure
Derived metrics
Abandon% = (SDK Init − SDK Callback) / SDK Init
Vault Miss% = isPanFetched=false / Output API event count
Backup Success% = of vault-miss users, % reaching KYC_PAN_VERIFIED
CB→Verify% = KYC_PAN_VERIFIED / SDK Callback
All %s cohort-correct: 7-day window from KYC_RISKDETAILS_SUBMIT
