Engineering Root Cause

KVO is Apple-only. Android uses a completely different bridging mechanism — no KVO means no cold-start crash on Android. This directly explains why 77% of failures are iOS even though Android has more users.
Issue 1 — KVO cold-start crash documented but never fixed

docs/prod-spec/biometric/issues/issue.md already describes this exact crash. React Native's bridge tries to register a KVO observer on RNBiometric module. On cold start, the observer sometimes attaches before the module is fully initialised — or a duplicate observer is registered — causing the module to be in a broken state when Face ID fires.

Fix deployed May 24 2026. iOS fail rate dropped from 75.2% pre-fix to 12.7% on Jun 1, confirming the KVO cold-start crash was the primary driver.
Issue 2 — RNBiometric.m has zero lifecycle management

The entire Objective-C bridge file is just method declarations — no dealloc, no observer setup/teardown:

// ios/Fin/RNBiometric.m — entire file @interface RCT_EXTERN_MODULE(RNBiometric, NSObject) RCT_EXTERN_METHOD(verifyBiometric: ...) RCT_EXTERN_METHOD(generateKeyPair: ...) // ... just declarations, nothing else @end // No dealloc. No observer setup. No initialization sequencing.
Issue 3 — Screen recording module had the same bug and was already fixed

In ios/Fin/AppDelegate.mm, the team hit this identical KVO problem on screen recording and fixed it:

// Observe screen capture state changes via NSNotification // (replaces crash-prone KVO on UIScreen) ← comment says it all [[NSNotificationCenter defaultCenter] addObserver:self selector:@selector(screenCaptureDidChange:) name:UIScreenCapturedDidChangeNotification object:nil];
The fix pattern already exists in this codebase. It just was never applied to RNBiometric.
Issue 4 — errorCode from native is available but never logged

RNBiometric.swift correctly returns the LAError code. But useBiometricLogin.ts discards it:

// RNBiometric.swift — returns errorCode correctly let laErr = authError as? LAError let errorCode = laErr?.code.rawValue ?? -1 resolve(["success": false, "errorCode": errorCode]) // useBiometricLogin.ts:128 — throws it away if (!biometricResult.success) { postEvent(BIOMETRIC_VERIFY_FAILED); // ← errorCode missing }

Fixes Required

  • P0
    Add dealloc + observer lifecycle to ios/Fin/RNBiometric.m

    Follow the same pattern already used in AppDelegate.mm for screen recording. Add explicit -dealloc to the Objective-C bridge to clean up KVO observers when the module is deallocated on cold start. This is the primary fix for the iOS spike.

  • P0
    Log errorCode in BIOMETRIC_VERIFY_FAILED Amplitude event

    One-line change in useBiometricLogin.ts:128:
    postEvent(BIOMETRIC_VERIFY_FAILED, { errorCode: biometricResult.errorCode, error: biometricResult.error })
    This is the fastest diagnostic — within a week of shipping, we'll know what % of failures are the KVO crash (errorCode -4) vs genuine Face ID misses.

  • P1
    Stagger dual push sends (coordinate with marketing)

    Currently RestB2C (~12k) + Addl (~29k) fire at the exact same timeslot — 41k simultaneous cold starts at 09:15 IST. Stagger by 15–30 min flattens the spike. Does not reduce total failures, only spreads them over time.
