❌ Scenario B — Partial funds split-panel High
Scenario B
The two-panel green/amber split screen (immediate units in green + GTC-queued units in amber) is a novel UI pattern. No Figma frame covers this. This is one of three core sell scenarios and its absence means engineering has no handoff material for ~30% of sell interactions.
❌ Scenario C — GTC queue sell initiation High
Scenario C
Blue panel + "Queue sell request" CTA for when AMPL has zero liquidity. No frame in Figma. This scenario requires a completely different visual tone (blue = patience) vs Scenarios A/B (green/amber). Not a minor omission — it's an entirely different user story.
❌ Near-cutoff amber nudge (3:00–3:30 PM IST only) Medium
Sell Screen
Conditional inline note: "Orders placed after 3:30 PM today will settle on T+2." Only shown during 3:00–3:30 PM window. Requires a time-conditional UI state. No frame or overlay designed for this.
❌ Error states — inline red error, network failure High
Sell Screen
PRD specifies: inline red error below CTA (not navigate away), explicit "not placed" copy on failure, "Your units are safe" fallback after 2 consecutive failures with support link. None of these states are designed. Critical for user confidence.
❌ Loading state — inline spinner on confirm CTA Medium
Sell Screen
PRD mandates inline spinner on confirm CTA (no full-screen loader). This micro-interaction prevents double-taps and feels native. Absent from Figma. Engineering will default to a full-screen loader without this spec.

Portfolio Card Sell Strip States
7 cohort-specific card states driven by the PRD color system
0 / 7 identifiable
❌ Blue cooling strip with 90-day progress bar High
Sell-Cooling
"61 of 90 days" progress bar on the holding card. Blue = patience. Transitions to green on Day 90. This is the primary trust-building mechanism during the holding period. No frame is labeled or identifiable as this state.
❌ Green sell-ready strip ("Sell →" CTA) High
Sell-A · A1
Day 90 color transition: blue strip → green strip with "Sell →" CTA. The most important state in the sell discovery journey. No Figma frame is named or can be identified as this state. Buy/Sell filter tags exist but the card strip itself is absent.
❌ Amber "Sell in progress" strip High
All post-sell
After sell order is placed, card shows amber strip: "Sell in progress · Funds expected [date]". This is the primary feedback mechanism between order placement and settlement. Entirely absent.
❌ Gray kill switch strip (no CTA) High
Kill switch
Admin-disabled sell state. Gray color, no CTA, no explanation. PRD is explicit: gray = only state where user has zero actionable path. No Figma frame exists for this.
❌ Amber external demat strip (Sell-B, "Open demat" CTA) High
Sell-B
Permanent sell block for pre-Mar 10 2026 holders using external demat. Amber strip with "Open demat" CTA to convert. No frame designed. This cohort likely represents a meaningful slice of early adopters.
❌ GTC expiry — portfolio card recovery state Medium
Scenario C
When GTC sell order expires, units are "returned" (PRD is explicit: never "cancelled"). Card reverts to green sell-ready state. No frame designed. Copy precision here is critical — "returned" vs "cancelled" affects user trust.
🟡 Buy/Sell filter tags on holdings list Low
Portfolio
Partially present: "tags_v3" component with Buy/Sell text is in the Figma. However, the card-level strip states that drive whether a bond shows "Sell →" are undesigned — so the tag alone can't communicate sell eligibility without the strip context.

Push Notification Arc
8 notification events across the full sell lifecycle
0 / 8 designed
❌ All 8 push notification templates missing Medium
Push
The PRD defines a complete communication arc:

• sell_approaching — Day 60 pre-unlock (blue)
• sell_unlocked — Day 90 unlock (green)
• sell_executed — Funds credited (green)
• sell_partial — Partial fill + GTC queued (amber)
• sell_queued — Full GTC queue (blue)
• sell_cap_lifted — AMPL capacity restored (green)
• sell_expired — GTC expired, units returned (blue → green)
• sell_order_held_rating — Rating downgrade hold (amber)

None have notification design frames in the Figma. Copy tone and color for each must match the PRD semantic system.
Design & Handoff Suggestions
A
Rename frames to reflect cohort state, not sequence
Frames named "Portfolio Mobile 16–53" are unhandable. Rename to e.g. Holdings / Cooling Period (Blue Strip, Day 61) or Holdings / Sell Ready (Green A1). Engineers cannot map by number alone.
B
Build a multi-state component for the sell strip
The portfolio card strip has 7+ distinct states (blue cooling, green ready, amber in-progress, amber external demat, gray kill switch, green funds credited, blue GTC queued). Make it one Figma component with named variants — not 7 separate frames.
C
Add a color semantics reference frame to Figma
The 4-color language (Green = ready, Amber = urgent, Blue = patience, Gray = kill switch) is non-negotiable per the PRD. Add a reference frame in Figma that documents this. Any designer or engineer joining later will break the system without it.
D
The buy flow is missing entirely — link or design it
Units Selection → Order Review → Payment are where sell eligibility is first surfaced to users. These aren't in the Portfolio Figma and there's no cross-file link. The "sell as a buy incentive" story can't be shipped without these 3 screens.
E
Fix post-sell copy to match PRD exactly
Figma shows "Sell order successful." PRD mandates "Sell order placed." Not cosmetic — "placed" sets accurate expectations (not yet settled), while "successful" implies immediate completion and could cause support tickets.
F
Design the sell-cashflow interaction explicitly
When a user sells mid-coupon cycle, what happens to upcoming coupon payments on the cashflow tab? The Figma shows cashflow for regular coupon bonds but the sell flow doesn't address this. Users will ask.
G
DDPI auth flow needs entry + return states
Multiple cohort states (A2) trigger "Authorise DDPI →". The Figma has only a Digio placeholder. The Aspero-side pre/post states (entering DDPI auth, returning from Digio success or failure) need designed transitions — otherwise A2 users hit a dead end.
H
Accessibility: add icons alongside color signaling
The PRD's 4-color system communicates state entirely through color — a WCAG accessibility violation. The PRD already uses "⏰" and "●" in copy. Confirm Figma bakes these icons into the strip designs, not just the text spec.
I
Scope the request-callback widget away from sell flows
The "Clear doubts over call!" widget is a general support touchpoint. Surfacing it mid-sell flow (especially Scenario C GTC queue) will cause abandonment. The PRD's error handling already includes a "Contact support" link — the callback widget placement relative to sell screens needs explicit scoping.
