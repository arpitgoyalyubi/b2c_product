CRITICAL Block dev start if unresolved
C1Holdings re-check at eSign is noted but not drawnCritical
Flowchart 2 has a note: "Holdings re-checked at eSign time to handle bond-maturity race condition." But Flowchart 3 (eSign flow) has no such gate between "User reviews form" and "Digio eSign." If a bond settles after Gate 4 passes but before the user signs, the system allows closure of a non-NIL account. This check must be an explicit step in the eSign flow with a failure path back to the Holdings Detected screen.
→ Add holdings re-check step to Flowchart 3 before Digio handoff
C2Gate 3 blocks re-initiation after CLOSURE_REJECTEDCritical
The pre-check gate redirects any existing request to the status screen. If the existing request is CLOSURE_REJECTED, the user should be allowed to re-initiate — not blocked. Gate 3 needs a sub-check: if existing request status = REJECTED, pass through to the closure form. Currently the flow treats PENDING and REJECTED identically.
→ Update Gate 3 logic: REJECTED state → pass through, PENDING → redirect
C3Bond matures during CLOSURE_PENDING — no flow existsCritical
A user submits with NIL balance → bond settles next day → demat now has holdings while status is CLOSURE_PENDING. No flowchart covers this. Two possible decisions: (a) ops can still approve and NSDL handles settlement, or (b) request auto-rejects with system reason. Neither path is drawn. This is the open question flagged in the PRD that is still unresolved.
→ Decision needed from DP Ops before design can be finalised
C4Consent checkbox and warning banner missing from eSign flowCritical
The mockup (Screen 2) correctly shows a consent checkbox and a warning banner about UCCC deactivation and profile removal. These also appear as node J in the Master Decision Tree. But Flowchart 3 (eSign detail) skips from "User reviews form" directly to Digio — the consent gate is absent. If the user dismisses the form without checking consent, should eSign still proceed?
→ Add explicit consent gate in Flowchart 3 before Digio handoff
SIGNIFICANT Resolve before UI handoff to engineering
S1No DP Ops notification mechanism shownSignificant
Flowchart 4 starts with "DP Ops opens Aspero Demat Closure collection" — but what triggers them? There is no email alert, Slack ping, in-admin badge, or daily digest shown. If 10 requests arrive overnight with no alert, TAT will slip. The trigger mechanism must be specified (real-time push notification, daily digest email, or dashboard badge count).
→ Specify Ops notification mechanism; add to ASP-4118 (admin list story)
S2eSign timeout / session expiry not handledSignificant
The eSign flow has three outcomes: Success, Failed/Error, Abandoned. A 4th path is missing: session expired (user takes too long, or Aadhaar OTP expires mid-flow). Digio sessions time out. The error message and retry path for expiry is different from a "failed" signature. This needs its own branch in Flowchart 3.
→ Add Session Expired branch to Flowchart 3 with retry copy
S3"Phase 2 available?" has no implementation specSignificant
Both the Master Decision Tree and Pre-Check Gate flow have a "Phase 2 available?" decision diamond. There is no spec for how this is evaluated — feature flag, release date, user cohort, or remote config. Engineering will ask this immediately on receiving the flows. Without a mechanism, this node is floating.
→ PM + Eng to confirm: feature flag (recommended), date-based, or cohort; document in PRD
S4Bulk approval not specified in ops flowSignificant
The admin UI (Screen 10) has bulk checkboxes and a "Download Selected / Bulk ZIP" action. Flowchart 4 only covers single-request approval/rejection. If ops bulk-approves 5 requests, does each trigger the full parallel action set (status change + soft-delete + UCCC API + emails) independently and simultaneously? Is there a confirmation modal? A separate ops sub-flow is needed.
→ Add bulk approval flow as a separate sub-flow in Flowchart 4 or a new Flowchart 7
S5CANCELLED state missing from the state machineSignificant
The status screen shows "Contact Support to cancel" (ASP-4126). If support cancels a CLOSURE_PENDING request, there is no CANCELLED state in Flowchart 6's state machine. Additionally, Gate 3 (existing request check) has no way to handle a cancelled request — should it pass through or block? The state machine and Gate 3 both need updating.
→ Add CANCELLED state to Flowchart 6; update Gate 3 to handle CANCELLED as "pass through"
MINOR / NOTES Document for engineering handoff
M1Re-entry login path unclear (same mobile number?)Minor
Flowchart 5 says fresh onboarding creates a new User ID, new demat, new UCCC, fresh KYC. But the PRD specifies email and mobile number are retained on soft-delete. The flow doesn't clarify whether the returning user logs in with the same mobile, or whether the auth layer sees the mobile as already registered. If the same mobile is retained but the User ID changes, the login layer needs special handling.
→ Clarify in re-entry flow: same mobile → detected at login → show re-entry popup
M2UCCC failure → email sequence is ambiguousMinor
In Flowchart 4, when the UCCC API fails (G2b), the "Approval NOT rolled back" path converges at H (statement email fired to user). A reader could interpret this as a bug — the email fires even when UCCC deactivation failed. This is likely intentional but needs an explicit note: "Statement email fires regardless of UCCC API outcome."
→ Add annotation to Flowchart 4 node H: email fires regardless of UCCC result
M3Email triggers not mapped to specific flow stepsMinor
ASP-4125 requires email templates for: (a) request submitted, (b) approved, (c) rejected, (d) account statement. The flowcharts mention emails at several nodes but there is no consolidated view of which template fires at which step. Not a blocker for the designer, but will become one during engineering handoff when email triggers need to be wired.
→ Add an email trigger map table to the PRD or as Flowchart 7
M4Flowchart 1 is too long to be a useful stakeholder artifactNote
The Master Decision Tree conflates pre-checks (Gates 1–4) and the closure ceremony (eSign, ops approval, parallel actions) into one long flow. Flowcharts 2–4 correctly break these out. For stakeholder reviews, Flowchart 1 should be simplified to a swim-lane summary referencing 2–4 for detail, not the single source of truth.
→ Designer: simplify Flowchart 1 to a 1-page swim-lane; keep Flowcharts 2–4 as the spec
