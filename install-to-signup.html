CTA% Drop · Root Cause Hypothesis Ranking

Repeat-user CTA% = 98% (logged-out users, same screen) — rules out technical glitch and UI bugs. Ad group gap: In-market audience 92% vs Bonds_UACe 37% on the same screen — confirms user intent quality as primary lever.

HypothesisStrengthKey Evidence
User intent quality degradation from spend mix Very Strong In-market 92% vs UACe 37% on same screen — 2.5× gap
Why it matters: CTA% is the primary bottleneck in this funnel. A 2.5× CTA% gap between ad groups on the identical screen isolates user intent as the lever — not the product, not the UI.
Supporting data: See Sections 7–10 · Ad Group tab — In-market shows ~92% CTA%; Bonds_UACe shows ~37%. All other variables (screen, device, time of day) held constant across both groups.
How to validate: Run a spend-shift experiment: move 20% of Bonds_UACe budget to In-market for 2 weeks. If blended CTA% rises proportionally to the spend share change, hypothesis confirmed.
Recommended action: Reallocate budget toward In-market + purchase-signal audiences. Cap broad UACe spend until creative/landing-screen alignment is fixed. Track blended CTA% weekly in Section 4.
Ad creative ↔ landing screen disconnect Strong UAC auto-creatives show generic investment imagery; screen is bonds-specific
Why it matters: Users arrive expecting a broad investment product (mutual funds, stocks) based on the ad creative, then see a bonds-specific screen. The cognitive mismatch causes immediate exit before engaging with the CTA.
Supporting data: Google UAC auto-generates creatives from app store assets — these are generic “grow your money” themes. The landing screen is explicitly about bonds (“Invest in Bonds”, SEBI badge, coupon rate language).
How to validate: Create a bonds-specific creative set (coupon rate, SEBI-regulated, ₹ denominations). Run A/B split vs. auto-creatives on same UACe campaign. Measure CTA% difference within same ad group.
Recommended action: Upload 5–7 bonds-specific creative variants to the UAC campaign. Include the phrase “bonds” in all headlines. Expected to recover 15–25pp of UACe CTA%.
“Bonds” category awareness gap Strong Repeat users 98% CTA% — they know the product; new broad cohorts don’t
Why it matters: The 98% repeat-user CTA% is a ceiling benchmark — it shows the screen itself is fine for informed users. The gap to new cohorts is purely a “what is this product?” awareness problem.
Supporting data: Repeat users (logged-out, same MOBILE_LAUNCH_VIEW screen) convert at 98%. New broad-match cohorts drop to ~37–45%. The screen does not explain bonds before asking the user to tap “Get Started.”
How to validate: Add a one-liner below the headline: “Bonds = fixed deposits with higher returns, SEBI-regulated.” Run A/B test. Measure CTA% lift for UACe / opentargeting segments specifically.
Recommended action: Add a 10-word bond explainer to the pre-signup screen. A/B test with Amplitude feature flag. Segment analysis by new vs. returning user to confirm lift is concentrated in new cohorts.
Missing social proof Moderate No “1L+ investors / ₹500cr” signal on launch screen
Why it matters: New low-intent users from broad campaigns need external validation (“others like me are using this”) before committing. Repeat users already have internal validation; this fix targets new cohorts only.
Supporting data: The MOBILE_LAUNCH_VIEW screen currently shows only product claims (returns, SEBI badge). No social proof signals (investor count, AUM deployed, reviews) are visible above the CTA fold.
How to validate: Add “1,00,000+ investors · ₹500Cr+ invested” badge near the CTA button. A/B test. Measure lift for new-user cohorts (specifically UACe and opentargeting segments).
Recommended action: Add a single social proof line above CTA: “Join 1L+ investors earning fixed returns.” Low engineering effort, measurable via Section 4 CTA% within 7 days.
Cold start render latency (Android Low) Moderate Largest tier ~23k LV; warm app = 98% CTR; cold start delays CTA render
Why it matters: Android Low is the single largest tier by LV volume (~23k). Even a 5–10pp CTA% recovery here adds meaningful absolute CTA volume vs. fixing a smaller tier.
Supporting data: Warm-app repeat users (98% CTA%) use faster cached renders. Cold start on low-RAM Android (<3GB RAM) can take 3–6 seconds. If the CTA button renders after the user’s patience window, they exit. See Section 4 — Android Low has the lowest CTA% of all mobile tiers.
How to validate: Profile cold-start time-to-interactive for Android Low devices in Firebase Performance. Compare CTA% segmented by session type (cold vs. warm) if that attribution is available.
Recommended action: Engineering: lazy-load non-CTA assets on cold start. Pre-render the CTA button above the fold before carousel images load. Set a performance budget: time-to-CTA-interactive < 1.5s on mid-tier Android.
Language / comprehension barrier Moderate Tier-2/3 expansion; screen is English-only; “SEBI Registered Broker” requires literacy
Why it matters: Budget expansion into tier-2/3 cities (visible in campaign data) brings users with limited English financial literacy. The screen’s credibility signals (“SEBI Registered Broker”, “fixed returns”) are English-only and require domain knowledge to interpret.
Supporting data: Google Ads geo distribution shows increasing spend in tier-2/3 cities. The MOBILE_LAUNCH_VIEW screen has no Hindi or regional language text. Android Low tier (likely highest tier-2/3 share) has the lowest CTA% among mobile tiers.
How to validate: Segment CTA% by device language setting (available in Amplitude as a user property). Compare Hindi-locale vs. English-locale new users on same ad group. If Hindi locale shows <20% CTA%, language barrier is confirmed.
Recommended action: Add Hindi subtitle under the main headline: “SEBI-registered platform — bonds give higher fixed returns than FDs.” A/B test on Android Low tier specifically. Low engineering effort.
Carousel confusion (2-slide ambiguity) Weak–Moderate Users may wait for more slides; hard to isolate without A/B
Why it matters: The screen has 2 carousel slides (returns-focused + security-focused). Users from carousel-heavy app stores may wait for more slides or perceive the CTA as scroll-dependent rather than always-visible.
Supporting data: No direct data isolating carousel confusion. Indirect signal: Views/User >1.5 in Section 5 could reflect users re-opening the screen and scrolling/swiping before tapping. Hard to separate from other re-engagement reasons.
How to validate: A/B test: single-slide static screen vs. 2-slide carousel. Amplitude will show CTA% difference immediately. If single-slide outperforms, carousel is confirmed as a confounding pattern.
Recommended action: Run A/B test (Feature Flag in Firebase Remote Config) before investing in carousel redesign. Low priority unless single-slide A/B shows >5pp lift.
Visual / UI issues (button, SEBI badge) Weak Repeat users 98% on same screen — rules out UI as primary driver
Why it matters: While UI quality matters in general, the 98% repeat-user CTA% on the same screen establishes that any UI imperfections are insufficient to block conversion for motivated users.
Supporting data: Repeat users (same MOBILE_LAUNCH_VIEW screen, same CTA button, same SEBI badge placement) convert at 98%. If UI were a significant barrier, repeat users would also be blocked. The gap is cohort-specific, not screen-specific.
How to validate: Not worth A/B testing until higher-priority hypotheses (intent quality, creative disconnect, awareness gap) are addressed. UI improvements can be shipped without measurement.
Recommended action: Deprioritize UI-only fixes. Focus engineering resources on cold-start latency (Hypothesis 5) and language barrier (Hypothesis 6) which have clearer conversion pathways.
Technical glitch Ruled out 98% repeat-user CTA% closes this completely
Why it matters: Technical glitches are always a first suspect in conversion drops, but they can be definitively ruled out by a high-performing baseline on the same screen.
Supporting data: Repeat users (logged-out, same MOBILE_LAUNCH_VIEW screen) fire CTA at 98%. If the button were broken, API were failing, or JS were erroring, repeat users would also fail. 98% is definitively above any plausible glitch threshold.
How to validate: Already validated — no further investigation needed. Monitor Firebase Crashlytics for regressions but don’t prioritize investigation.
Recommended action: No action. Confirm with on-call that no recent deploy changed the launch screen. Then redirect attention to Hypotheses 1–3.
